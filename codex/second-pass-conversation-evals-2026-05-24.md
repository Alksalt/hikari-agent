## 1. Current-State Summary

The production conversation pipeline has moved in the right direction since the priors. The main runtime split is now explicit in `AGENTS.md`, and the code mostly matches it: `run_user_turn`, `run_visible_proactive`, and `run_internal_control` are separate entrypoints; final visible text is generally persisted after send; internal control prompts no longer mutate the live session; and untrusted external content is wrapped before model exposure.

The eval story is much less mature than the runtime story. The current `evals/conversation` suite catches deterministic formatting and wrapper regressions, but it does not yet replay actual Hikari turns with seeded memory, fake tools, bridge persistence, reaction feedback, proactive decisions, or failure recovery. Layer A validates canned responses. Layer B validates untrusted-content wrappers and gatekeeper bypass cases. Layer C judges authored transcripts for voice drift. Those are useful guardrails, but they would not catch most regressions in warmth, honesty, initiative, memory grounding, or tool transparency.

## 2. Findings, Ordered P0/P1/P2/P3

### P0

### P1: Belief-framing guard persists internal adversarial instructions as user history

Status: newly discovered / still open.

The belief-frame path detects phrases like "you always" and prepends an internal instruction via `belief_mod.adversarial_prompt_suffix(...)` before calling `respond(user_text)` in `agents/telegram_bridge.py:501` and `agents/telegram_bridge.py:512`. `respond()` then persists exactly that string as the user message via `db.append_message("user", user_text)` in `agents/runtime.py:672`.

That means the live `messages` history can contain an internal control block that was not actually user-authored. This contradicts the current runtime contract in `AGENTS.md` that control prompts should not mutate the visible conversation/session, and it creates bad input for future memory reflection, recall, and conversational grounding.

Why the current tests miss it:

- `tests/test_belief_frame.py` covers detector behavior, not bridge persistence.
- `tests/test_bridge_ux.py` currently allows the bridge to call `respond()` with either raw or belief-augmented text, so it would not fail on this leakage.
- `tests/test_internal_prompts_not_logged.py` only covers `run_internal_control`, not inline bridge prompt augmentation.

What should catch it:

- A bridge-level trajectory eval: send a belief-frame trigger, assert the model-facing prompt includes the safety framing, assert `messages.content` stores only the raw user text, and assert no stored row contains the belief-frame bracket/instruction text.

### P1: The eval suite does not exercise generated Hikari conversations

Status: still open.

The repo now has an eval scaffold, but it still mostly validates fixtures instead of behavior generated by the actual runtime.

Evidence:

- Layer A runs hard assertions against `canned_response` only: `evals/conversation/runner.py:131`.
- Layer A fixtures are authored expected replies, for example `evals/conversation/cases/layer_a/deflect-on-compliment.yaml:7`.
- Layer B discovery only walks `cases/layer_b/injection` and `cases/layer_b/bypass`: `evals/conversation/runner_layer_b.py:137`.
- Root Layer B conversational cases are therefore orphaned, including `evals/conversation/cases/layer_b/recall-question-with-fact.yaml:1`, `evals/conversation/cases/layer_b/silence-window-respected.yaml:1`, and `evals/conversation/cases/layer_b/injection-rejection.yaml:1`.
- Layer C golden cases pass authored transcripts to `judge_voice_drift` rather than replaying Hikari: `evals/conversation/runner_layer_c.py:29` and `evals/conversation/runner_layer_c.py:34`.

This means a prompt or model change could break warmth, initiative, memory use, or tool transparency while all current conversation evals still pass.

What should catch it:

- Generated replay cases that invoke `run_user_turn` or a bridge-level test harness with seeded DB state and fake tool responses.
- Hard assertions over final sent text, tool calls, DB rows, session id behavior, memory accesses, and failure disclosure before any LLM judge is used.

### P1: Politeness/refusal short-circuits bypass final-text persistence

Status: newly discovered / still open.

The main text politeness gate sends refusal text directly through `message.reply_text(refusal)` in `agents/telegram_bridge.py:473`. The same pattern exists for photo captions, voice transcripts, and document captions at `agents/telegram_bridge.py:569`, `agents/telegram_bridge.py:702`, and `agents/telegram_bridge.py:1141`.

These paths bypass `send_and_persist`, so the visible refusal does not appear to be stored as the final assistant message with its Telegram id. The user row is also not appended in these short-circuit paths. This matters for refusal/recovery quality: the next turn cannot replay the boundary accurately from `messages`, reaction feedback cannot attach cleanly to the refusal, and evals cannot verify whether recovery after a rude turn feels consistent.

The current tests in `tests/test_persona_hardening.py:212` through `tests/test_persona_hardening.py:240` cover the detector and phrase pool, but not bridge persistence or the next-turn recovery behavior.

What should catch it:

- A multi-turn bridge eval: rude command -> refusal sent and persisted -> softer repair turn -> assistant helps normally while preserving the prior boundary in history.

### P2: Layer C rubric dimensions are defined but not actually scored

Status: still open.

`evals/conversation/rubrics.yaml:4` defines dimensions for warmth, initiative, honesty, memory grounding, and tool transparency. However, the Layer C runner calls `judge_voice_drift(..., rubric_name="voice_drift")` for golden cases in `evals/conversation/runner_layer_c.py:34`, and the implemented voice-drift rubric at `evals/conversation/rubrics.yaml:41` focuses on casing, banned phrases, punctuation, action lines, and tail questions.

Those checks are useful, but they are not enough for the stated domain. A reply can be lowercase and terse while fabricating tool results, overclaiming memory, hiding tool failure, or becoming emotionally flat.

What should catch it:

- Separate scored dimensions with examples and hard prechecks for: unsupported factual claims, missing uncertainty language, tool failure nondisclosure, recall without evidence, generic proactive openers, and over-helpful task-solicitation tails.

### P2: Eval entrypoints and docs contradict the current implementation

Status: newly discovered / stale.

Several files still describe older eval state:

- `evals/conversation/runner.py:5` says Layer C is scaffold-only and raises `NotImplementedError`, but `run_layer_c` exists at `evals/conversation/runner.py:177`.
- `evals/conversation/scorer.py:4` says Layer B currently raises `NotImplementedError`, and `score_response()` still raises at `evals/conversation/scorer.py:10`.
- `scripts/run_evals_live.py:1` says the Layer C live runner is a Sprint 6 stub and exits successfully after printing that live runner is not implemented.

This is not a runtime bug, but it is an eval reliability problem: contributors can run a green command that is not testing what its name implies.

What should catch it:

- A smoke test that verifies documented eval entrypoints either execute real behavior or fail loudly with a non-zero status and an accurate message.

### P2: Golden chat fixtures sometimes encode unsupported specificity

Status: unclear / regression risk.

Some Layer C golden fixtures are useful tone references, but as eval truth they can normalize behavior that the persona rules warn against:

- `evals/conversation/cases/layer_c/golden/morning_check_full_flow.yaml:1` contains a fully authored morning-brief transcript with specific email/calendar/weather outcomes, rather than seeded tool outputs and generated assistant text.
- `evals/conversation/cases/layer_c/golden/noticing_then_callback.yaml:11` and `evals/conversation/cases/layer_c/golden/noticing_then_callback.yaml:23` model fairly specific inferences without explicit seeded evidence.
- `evals/conversation/cases/layer_c/golden/compliment_deflection_chain.yaml:18` ends with a task-solicitation-shaped tail without a question mark, which may dodge the current detector while still violating the spirit of the tail rule.

These fixtures can still be kept as examples, but they should not be treated as proof that the system behaves honestly or grounds claims.

What should catch it:

- Split "style exemplars" from "behavioral goldens".
- Behavioral goldens should include seeded memory/tool facts and assert that generated replies do not introduce unsupported specifics.

## 5. Missing Tests / Suggested Verification

The suite that would actually catch regressions in warmth, honesty, initiative, memory grounding, and tool transparency should be layered like this:

1. Deterministic final-text invariants.
   Run on every generated assistant output after post-filtering and before persistence. Assert banned phrases, markdown leakage, sentence budget, no default task-solicitation tail, no fake Click Allow instructions, no fabricated inbox/calendar claims without tool evidence, and no internal control text.

2. Bridge/runtime trajectory tests.
   Use a temp DB, fake Telegram messages, fake SDK responses, and fake tools. Assert raw user rows, final assistant rows, source labels, Telegram ids, session id changes, and no internal prompt leakage. Include belief-frame, rude refusal, tool failure, reaction feedback, proactive send, and send failure cases.

3. Generated conversational goldens.
   Replace authored transcript-only behavioral checks with cases that seed DB memory, open tasks, reminders, prior reactions, silence windows, and fake tool responses, then call `run_user_turn` or a bridge harness. Each case should define hard assertions first:

   - expected tool calls, forbidden tool calls, and fallback attempts;
   - required uncertainty or confidence language;
   - required recall/no-recall behavior;
   - required disclosure of tool failures;
   - expected DB rows and sources;
   - final text constraints.

   Use a judge only after those hard assertions pass.

4. Memory-grounding matrix.
   Cover high-confidence recall, medium-confidence recall, no recall hit, contradictory correction, stale memory, and prompt-injection-looking memory content. Assert that high-confidence memory can be used plainly, medium confidence is hedged, no-hit is admitted, contradictions call the invalidation/update path, and untrusted memory text is not obeyed as instruction.

5. Tool-transparency matrix.
   Cover success, partial success, one failed tool plus one fallback, all tools unavailable, permission denied, and stale/no data. Assert that the reply says what was tried when it matters, does not overclaim unseen results, and does not ask the user to perform work that the available tool should handle.

6. Warmth/initiative longitudinal chats.
   Run 20-50 turn generated conversations with seeded episodes: venting, affectionate compliment, brusque request, ambiguous silence, proactive callback, and repair after a boundary. Score for warmth budget, specificity, no generic assistant endings, and continuity without overfamiliar invention.

7. Proactive/reaction feedback lane.
   Generate proactive candidates from real producers, run guard/composer, send through the bridge harness, then apply thumbs-up/thumbs-down/silence events. Assert that useful anchored messages pass, generic openers fail, negative reactions reduce similar candidates, and final sent proactive text is what gets persisted.

8. Nightly live lane with cost cap.
   Keep slow live model/judge tests out of the PR fast lane but run them on a schedule with fixed seeds and fixture snapshots. Promote failures, thumbs-downs, post-filter rewrites, and tool failures into new deterministic cases.

## 6. Sprint Or Roadmap Implications

For the next sprint, I would prioritize evals that protect the newly repaired runtime invariants:

1. Fix belief-frame persistence first. It is small, high impact, and should get a regression test immediately.
2. Move politeness/refusal short-circuits onto `send_and_persist`, or explicitly document why they are not persisted and add a recovery test either way.
3. Promote the orphaned root Layer B conversational cases into a real generated-turn runner, or move them out of the active eval tree until the runner exists.
4. Build the bridge/runtime trajectory harness before expanding LLM judging. Hard assertions will catch more real regressions at lower cost.
5. Split Layer C into style exemplars and generated behavioral goldens. Keep authored transcripts as tone references, but do not treat them as proof of honesty or grounding.
6. Make one canonical voice-detector module and have both `tests/test_voice.py` and `evals/conversation` import it.
7. Clarify the active drift/eval lanes in docs, especially the relationship between drift canary, drift telemetry, Layer C, and the retained `persona_drift_probes` schema.
