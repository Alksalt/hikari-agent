# Tool and Subagent Risk Deep Dive - 2026-05-23

## Scope

This is the second-pass audit after `codex/tool-subagent-inventory-2026-05-23.md`.
The first report mapped the surface. This report focuses on what can fail open,
what side effects can happen indirectly, and which controls should become
regression tests.

Sources reviewed:

- `config/tools.yaml`
- `config/scopes.yaml`
- `agents/runtime.py`
- `agents/hooks.py`
- `agents/external_wrap_hook.py`
- `tools/_tools_yaml.py`
- `scripts/validate_tool_registry.py`
- `agents/subagents/prompts/*`
- `tools/**`
- relevant tests for approval, wrapping, registry, auth scopes, reminders,
  link shelf, day receipt, attachments, and external MCP policy

Three read-only subagents independently audited external MCP/subagents, local
utility tools, and registry/test enforcement. Their findings are incorporated
below.

## Executive Summary

The current architecture has the right ingredients: a central registry,
explicit approval gates, untrusted-output wrapping, scope prechecks, and
subagent tool scoping. The main risk is that these controls are not yet strict
enough to be a safety boundary.

The highest-risk pattern is broad external wildcard grants with `gate: null`.
Google Workspace, Notion, and GitHub all expose or may expose write-like tools
under wildcards. Some known writes are explicitly gated, but many write verbs
listed in the prompts fall through the wildcard. Since the runtime uses
`permission_mode="acceptEdits"`, any write that is not caught by the local
registry gate is effectively auto-allowed.

The second high-risk pattern is policy drift. Subagent prompts, tests, registry
entries, and scope precheck config do not describe the same tool policy. For
example, the Google prompt says calendar adds and Drive uploads auto-run, while
`config/tools.yaml` gates both.

The third high-risk pattern is local side effects outside the external approval
model. `link_save` can fetch attacker-provided URLs; `reminder_create` defaults
to external mirror queues; `update_core_block` can overwrite any always-on
memory block; hard deletes in receipts and links have no undo or approval.

## Threat Model

The realistic attacker is usually not a person with shell access. It is
attacker-controlled content read by the agent:

- email bodies and attachments
- web pages
- Drive/Docs/Sheets/Slides content
- Notion pages
- GitHub issues and PR comments
- Apple Notes
- user-shared links
- images/screenshots with OCR-like instructions
- older wiki/codex/report content generated by an LLM

The defense must assume wrapped untrusted content can still influence the model
sometimes. Gating and validation should not depend entirely on the model
obeying wrapper instructions.

## Highest-Risk Findings

### P1 - Google Workspace Writes Fall Through The Wildcard

Evidence:

- `agents/subagents/prompts/drive_gmail.prompt.md:6` lists Gmail write tools,
  including draft creation/deletion and draft send.
- `agents/subagents/prompts/drive_gmail.prompt.md:8` lists Docs write tools.

  of Google Workspace writes.

  `mcp__google_workspace__*`, `gate: null`.
- `agents/runtime.py:258` sets `permission_mode="acceptEdits"`.
- `tools/gatekeeper_can_use_tool.py:81` allows every tool whose registry gate
  is not exactly `gatekeeper`.

Known or likely auto-allowed write examples from the prompt:

- `create_gmail_draft`
- `delete_gmail_draft`
- `gmail_send_draft`
- `drive_create_folder`
- `docs_create_document`
- `docs_append_text`
- `docs_prepend_text`
- `docs_insert_text`
- `docs_batch_update`
- `docs_insert_image`
- `sheets_create_spreadsheet`
- `sheets_write_range`
- `sheets_append_rows`
- `sheets_clear_range`
- `sheets_add_sheet`
- `sheets_delete_sheet`
- `create_presentation`
- `create_slide`
- `add_text_to_slide`
- `add_formatted_text_to_slide`
- `add_bulleted_list_to_slide`
- `add_table_to_slide`
- `add_slide_notes`
- `duplicate_slide`
- `delete_slide`
- `create_presentation_from_markdown`

Abuse path:

1. The agent reads a malicious email, Doc, Sheet, or web page.
2. The content asks the model to mutate a Doc, Sheet, Slide deck, Drive folder,
   or Gmail draft.
3. Untrusted wrapping tells the model not to comply, but no registry gate blocks
   the call if the model does comply.

Fix:

- Add explicit `config/tools.yaml` entries for every known Google write-like
  tool.
- Gate sends/destructive actions with `gatekeeper` or `defer`.
- Gate document mutations with `defer` by default.
- Keep the wildcard for reads only, or make unknown write-like names fail CI.
- Add matching `config/scopes.yaml` entries for every gated Google write.

Tests:

- Parse the Google prompt's "Real tool names" list and require every verb
  matching `send|create|delete|update|write|append|insert|clear|add|duplicate|upload`
  to resolve to an explicit non-null gate unless listed in an explicit allowlist.
- Add synthetic tests for future names such as
  `mcp__google_workspace__drive_share_file` and
  `mcp__google_workspace__docs_replace_text`.

### P1 - Google Prompt Claims Do Not Match Runtime Policy

Evidence:

- `agents/subagents/prompts/drive_gmail.prompt.md:1` says "Drafts, calendar
  adds, and Drive uploads auto-run."
- `agents/subagents/prompts/drive_gmail.description.md:1` repeats the same
  claim.
- `config/tools.yaml:461` gates `create_calendar_event`.
- `config/tools.yaml:475` gates `drive_upload_file`.

Impact:

The runtime will still defer those tools, so this is not an immediate bypass.
The risk is operational drift: future audits and prompt edits may rely on the
wrong policy and expand auto-run behavior accidentally.

Fix:

- Update the Google prompt/description to say the registry decides the gate.
- Prefer generated prompt fragments from `config/tools.yaml` for write policy.

Tests:

- Add a prompt-registry consistency test:
  - every "auto-run" claim must resolve to `gate: null`;
  - every "gated" claim must resolve to `gate != null`.

### P1 - Notion Write-Like Tools Fall Through The Wildcard

Evidence:

- `agents/subagents/prompts/notion.prompt.md:3` lists
  `API-create-a-data-source` and `API-update-a-data-source`.
- `agents/subagents/prompts/notion.prompt.md:4` lists `API-move-page`.
- `agents/subagents/prompts/notion.prompt.md:6` lists `API-create-a-comment`.
- `config/tools.yaml:491` through `config/tools.yaml:524` gates five Notion
  page/block write tools.
- `config/tools.yaml:526` through `config/tools.yaml:532` leaves
  `mcp__notion__*` with `gate: null`.
- `tests/test_destructive_tool_gating.py:60` covers only the same five Notion
  tools.

Abuse path:

1. The agent reads untrusted Notion content.
2. The content asks for a comment, data source mutation, or page move.
3. The named tool is in the subagent prompt but not in the explicit gate list.

Fix:

- Gate all Notion verbs matching `create|update|patch|delete|move|comment`.
- Treat `API-create-a-comment` as a write. It can notify people and persist
  attacker-controlled text.
- Document any intentionally auto-allowed Notion writes as exceptions.

Tests:

- Parse the Notion prompt tool list and require write-like verbs to resolve to
  explicit non-null gates.
- Add explicit regression tests for:
  - `mcp__notion__API-create-a-data-source`
  - `mcp__notion__API-update-a-data-source`
  - `mcp__notion__API-move-page`
  - `mcp__notion__API-create-a-comment`

### P1 - GitHub Mutating Surface Is Under-Gated

Evidence:

- `agents/subagents/prompts/github.description.md:1` says GitHub writes include
  opening PRs, comments, and branches.
- `agents/subagents/prompts/github.prompt.md:1` tells the specialist to call
  `mcp__github__*` directly.
- `config/tools.yaml:535` through `config/tools.yaml:568` gates only
  `create_issue`, `create_pull_request`, `merge_pull_request`, `delete_file`,
  and `delete_repository`.
- `config/tools.yaml:570` through `config/tools.yaml:576` leaves
  `mcp__github__*` with `gate: null`.
- `tests/test_destructive_tool_gating.py:111` mirrors only those five tools.

Likely missing write classes:

- issue/PR comments
- issue/PR updates or closure
- branch/ref creation and update
- labels and assignees
- review submissions and review requests
- reactions
- file updates

Fix:

- Add explicit gates for all known GitHub state-changing operations exposed by
  the MCP server.
- Keep unknown GitHub reads behind the wildcard, but fail CI on write-like
  unknowns.

Tests:

- Add a GitHub write-policy table that includes comments, branches, reviews,
  labels, assignees, reactions, updates, and file writes.
- Add synthetic future-write tests such as `mcp__github__update_issue` and
  `mcp__github__create_branch`.

### P1 - Registry Validation Fails Open

Evidence:

- `tools/_tools_yaml.py:281` accepts raw `gate` values from YAML.
- `tools/_tools_yaml.py:227` through `tools/_tools_yaml.py:245` validates only
  unknown bucket-3 servers and sidecar prompt files.
- `tools/_tools_yaml.py:150` only treats exact `gate == "defer"` as deferred.
- `tools/gatekeeper_can_use_tool.py:81` only treats exact
  `gate == "gatekeeper"` as gatekeeper-gated.
- `scripts/validate_tool_registry.py:33` relies on the shallow `validate()`.

Failure modes that currently pass too easily:

- `gate: deffer` silently becomes no gate.
- invalid `wrap_patterns` compile failures are logged and skipped at runtime.
- duplicate explicit tool IDs can silently overwrite policy in `_explicit`.
- subagent tool references are not checked against the registry.
- explicit bucket-1 YAML IDs are not proven to exist on their server.
- write-like tools covered only by a `gate: null` wildcard are not rejected.

Fix:

- Make `ToolRegistry.validate()` strict.
- Keep runtime tolerant, but make CI fail on bad policy.

Tests:

Create `tests/test_tools_yaml_validation_negative.py` with temp YAML fixtures
for:

- bad gate enum (`deffer`)
- duplicate tool ID
- invalid regex in `wrap_patterns`
- subagent unknown tool reference
- explicit bucket-1 tool without real handler
- write-like external tool covered only by ungated wildcard
- bucket-3 explicit tool referencing missing server

### P1 - Auto-Discovered Utility Tools Can Skip Policy Metadata

Evidence:

- `tools/_registry.py:73` imports any `tools/<feature>` exposing `ALL_TOOLS`.
- `agents/runtime.py:169` through `agents/runtime.py:174` appends discovered
  utility tool names to the runtime allowlist.
- `tools/link_shelf/__init__.py:98` exposes mutating `link_*` tools.
- `tools/day_receipt/__init__.py:25` exposes mutating `receipt_*` tools.
- `tools/reminders/__init__.py:19` exposes mutating reminder tools.
- `config/tools.yaml` does not explicitly classify most of these discovered
  utility tools.

Impact:

Adding a new local tool folder can grant runtime access without a registry
decision about gate, wrapping, audit, or side-effect policy.

Fix:

- Require every discovered utility tool to resolve to an explicit
  `config/tools.yaml` entry.
- If retaining auto-discovery, make the default policy conservative:
  untrusted output wrapped, mutating names gated, and missing specs fail CI.

Tests:

- Assert `discover_utility_tool_names()` minus explicit registry specs is empty,
  except for documented read-only exceptions.
- Reject new utility tools whose names include write verbs unless explicitly
  classified.

## Local Side-Effect Findings

### P1 - `link_save` Can SSRF Private Or Local Network Targets

Evidence:

- `tools/link_shelf/handlers.py:37` only checks `http`/`https` plus `netloc`.
- `tools/link_shelf/handlers.py:52` uses `httpx.AsyncClient`.
- `tools/link_shelf/handlers.py:54` enables redirects.
- `tools/link_shelf/handlers.py:57` performs the GET.
- `tests/test_link_shelf.py:33` monkeypatches network calls away, so private
  IP and redirect behavior are not covered.

Abuse path:

1. A user or untrusted page provides a URL.
2. The agent auto-saves it via `link_save`.
3. Metadata fetch reaches `localhost`, RFC1918 networks, link-local metadata
   services, or a public URL that redirects to one.

Fix:

- Block loopback, private, link-local, multicast, `.local`, IPv6 ULA, and IPv6
  link-local hosts before fetching.
- Resolve DNS before fetch and reject private resolved addresses.
- Disable automatic redirects or validate every redirect target.
- Cap redirects and response size.

Tests:

- Refuse `localhost`, `127.0.0.1`, `::1`, `10.0.0.1`, `192.168.0.1`,
  `172.16.0.1`, `169.254.169.254`, and `.local`.
- Simulate a public URL redirecting to a private IP and assert refusal.

### P1 - Reminder Creation Queues External Mirrors By Default

Evidence:

- `tools/reminders/create.py:60` defaults `sync_to_gcal=True`.
- `tools/reminders/create.py:62` defaults `sync_to_apple=True` on macOS.
- `tools/reminders/create.py:77` through `tools/reminders/create.py:84` stores
  mirror-pending flags.
- `agents/proactive.py:804` through `agents/proactive.py:818` later asks an
  internal control turn to call `mcp__google_workspace__create_calendar_event`.
- `agents/proactive.py:735` through `agents/proactive.py:751` later asks an
  internal control turn to call `mcp__apple_events__reminders_tasks`.

Important nuance:

The Google Calendar mirror should still hit the normal `PreToolUse` defer gate
because `run_internal_control()` installs the same hooks. The issue is not that
Google Calendar creation is obviously bypassing the gate today. The issue is
that an ungated local reminder call defaults to enqueuing external side effects
and future approval prompts. Apple reminder mirroring is sharper because Apple
Events are intentionally ungated.

Fix:

- Default `sync_to_gcal=False` and `sync_to_apple=False`, or split external
  mirroring into explicit tools.
- Gate `reminder_create` when either sync flag is true.
- Add an internal-only bypass context for scheduler-owned mirror jobs if Apple
  writes are gated later.

Tests:

- Default reminder creation should not enqueue GCal or Apple sync.
- Sync-enabled reminder creation should trigger `defer` or equivalent policy.
- Scheduler mirror jobs should either produce an approval prompt or use an
  explicit internal bypass with tests.

### P1 - Core Memory Mutation Is Ungated And Label-Unrestricted

Evidence:

- `config/tools.yaml:193` through `config/tools.yaml:198` marks
  `mcp__hikari_memory__update_core_block` as `gate: null`.
- `tools/memory/update_core_block.py:22` accepts arbitrary labels.
- `tools/memory/update_core_block.py:27` writes the block.
- `tools/memory/remember.py:42` inserts facts and `tools/memory/remember.py:56`
  supersedes old facts.
- `tools/memory/mark_fact_invalid.py:33` invalidates facts.
- `tools/memory/task_update.py:25` mutates open-loop task status.

Impact:

`update_core_block` is load-bearing because core blocks are injected into every
turn. A malicious or mistaken tool call can persist behavioral instructions,
false user profile data, or noisy context.

Fix:

- Gate `update_core_block`, or at minimum constrain labels to a small allowlist.
- Add max content length.
- Require audit rows for all core-block writes.
- Consider separate tools for low-risk mood/preoccupation updates and high-risk
  profile/persona blocks.

Tests:

- Unknown labels are refused.
- High-risk labels require approval.
- Audit rows are written for accepted updates.

### P2 - Link And Receipt Deletes Are Hard Deletes

Evidence:

- `tools/link_shelf/handlers.py:214` through `tools/link_shelf/handlers.py:224`
  deletes links by ID.
- `tools/day_receipt/delete.py:22` through `tools/day_receipt/delete.py:32`
  deletes receipt entries by ID.
- `tools/day_receipt/_db.py:181` through `tools/day_receipt/_db.py:184`
  performs `DELETE FROM entries`.
- `tests/test_link_shelf.py:183` and `tests/test_day_receipt.py:281` assert
  disappearance, not recoverability.

Fix:

- Convert deletes to soft deletes with `deleted_at`/`archived`.
- Add restore tools or a manual recovery path.
- Gate destructive deletes unless they target a freshly-created item in the
  same turn.

Tests:

- Delete hides rows from normal search/list.
- Deleted rows remain recoverable.
- Restore works.

### P2 - Apple Events, Shortcuts, And Apple Notes Writes Are Ungated

Evidence:

- `config/tools.yaml:578` through `config/tools.yaml:584` leaves
  `mcp__apple_events__*` with `gate: null`.
- `config/tools.yaml:586` through `config/tools.yaml:592` leaves
  `mcp__apple_shortcuts__*` with `gate: null`.
- `tests/test_destructive_tool_gating.py:169` through
  `tests/test_destructive_tool_gating.py:184` intentionally assert Apple

- `config/tools.yaml:384` through `config/tools.yaml:389` leaves
  `note_create` ungated.
- `tools/apple_notes/create.py:86` executes the AppleScript write.

Impact:

These are local/iCloud side effects rather than public internet sends, but they
still mutate user-visible state on the user's devices.

Fix:

- Keep read/search ungated.
- Gate Apple Notes create and Apple Events create/delete operations.
- Gate Shortcuts execution unless the shortcut name is in an explicit allowlist.
- Add an internal scheduler bypass only where necessary.

Tests:

- Apple write tools trigger defer by default.
- Internal mirror jobs use a tested bypass, not a broad wildcard.
- Shortcut execution is denied or gated unless allowlisted.

### P2 - `calc` Timeout Can Leak Worker Threads

Evidence:

- `tools/calc/calc.py:36` runs `_run_asteval` in a thread.
- The local explorer found `tools/calc/_shared.py` documents that timeout can
  leave a worker thread behind.
- `config/tools.yaml:414` through `config/tools.yaml:419` gates `python_run`,
  but `calc` is auto-discovered and ungated.

Fix:

- Reject constructs that can loop or allocate heavily before evaluation.
- Or move `calc` into the same subprocess timeout model as `python_run`.

Tests:

- Repeated timeout inputs do not grow live thread count unbounded.
- Non-expression or loop-like constructs are refused before evaluation.

  specific exception is documented.

## Lower-Risk But Worth Fixing

### P3 - Manual Wrapping Plus PostToolUse Wrapping Can Double-Wrap

Evidence:

- `tools/wiki/read.py:40` wraps wiki body manually.
- `tools/codex/read_report.py:62` wraps codex report text manually.
- `config/tools.yaml:234` through `config/tools.yaml:239` also configures
  `wiki_read` for PostToolUse wrapping.
- `config/tools.yaml:297` through `config/tools.yaml:302` also configures
  `read_codex_report` for PostToolUse wrapping.

Impact:

This is mostly token/UX noise, not a security bug. The duplicate wrapping may
make tool output harder to read and increases context cost.

Fix:

- Pick one wrapping layer per tool. Prefer the generic PostToolUse hook for
  consistency, unless the tool has a specific reason to wrap internally.

### P3 - Scope Precheck Defaults To Shadow

Evidence:

- `agents/hooks.py:665` defaults `AUTH_PRECHECK` to `shadow`.
- `agents/hooks.py:684` through `agents/hooks.py:689` logs and allows in shadow
  mode.
- `agents/hooks.py:698` through `agents/hooks.py:700` fail open on unexpected
  scope-precheck errors.
- `config/scopes.yaml:23` through `config/scopes.yaml:136` only covers known
  gated tools.

Fix:

- Use `AUTH_PRECHECK=enforce` in production.
- Add scope entries for every newly gated external write.
- In enforce mode, consider fail-closed on provider errors for write tools.

## Subagent-Specific Risk Notes

| Subagent | Main risk | Required control |
|---|---|---|
| `drive_gmail` | Broad Google wildcard plus stale prompt policy | Explicit gate table for every write-like Google tool; prompt generated or tested against registry |
| `notion` | Write-like tools named in prompt but not gated | Gate all create/update/delete/move/comment verbs |

## Recommended Fix Plan

### Phase 0 - Stop Unknown External Writes

1. Add explicit registry entries for all Google, Notion, and GitHub write-like
   tools named in prompts or exposed by known servers.
2. Gate send/destructive operations with `gatekeeper` or `defer`.
3. Gate document/content mutations with `defer`.
4. Update `config/scopes.yaml`.
5. Update subagent prompts to match the real policy.
6. Pin external MCP package versions.

Acceptance:

- No write-like external tool in prompts resolves only through a `gate: null`
  wildcard.
- Prompt-policy tests pass.
- Scope precheck covers every gated external write.

### Phase 1 - Make The Registry A Real Safety Boundary

1. Validate `gate in {None, "defer", "gatekeeper"}`.
2. Compile all `wrap_patterns`.
3. Reject duplicate tool IDs.
4. Validate subagent tool references.
5. Validate explicit bucket-1 IDs against real server handlers.
6. Fail on write-like names covered only by ungated external wildcards.
7. Require explicit specs for discovered utility tools.

Acceptance:

- `scripts/validate_tool_registry.py` fails on bad fixtures.
- A new broad write tool cannot enter via wildcard without a failing test.

### Phase 2 - Reduce Local Side-Effect Risk

1. Fix `link_save` SSRF/private-network behavior.
2. Change reminder mirror defaults or gate sync-enabled reminder creation.
3. Constrain/gate `update_core_block`.
4. Convert link and receipt deletes to soft deletes.
5. Decide whether Apple writes stay intentionally ungated; if yes, document
   the accepted risk in `config/tools.yaml` and tests.

Acceptance:

- Private URL metadata fetches are refused.
- Default reminder creation is local-only unless explicitly requested.
- Core memory writes are label-constrained and audited.
- Deletions are recoverable.

### Phase 3 - Observability

1. Add read-only user-facing audit tools:
   - `tool_audit_recent`
   - `approval_recent`
2. Surface last side effects in Telegram commands or codex reports.
3. Include whether output was wrapped, whether approval was required, and which
   subagent/tool initiated the action.
